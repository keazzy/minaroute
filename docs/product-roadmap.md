## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can run the app and demo what you've built.
2. **Infrastructure before features.** Phase 0 sets up everything so feature work is fast and clean.
3. **Magic moment first.** The primary magic moment — a pilgrim feeling "I've got this" before travel — works by the end of Phase 1. The on-ground guidance moment works by the end of Phase 3.
4. **Test as you go.** Every task includes a verification step. Test the core path in **airplane mode** — offline is the default, not a fallback.
5. **Progressive enhancement.** Ship the simplest working version, then layer polish. The app must be fully usable **signed-out**; cloud sync is an enhancement.
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review it before starting the next phase.
7. **Correctness is sacred.** No rite content ships without a scholar-source citation. Content validation must fail the build if a citation is missing.
8. **Solo-buildable.** Tasks are scoped for one Claude Code session (~15–45 min) so a part-time founder can make steady progress.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the top of each phase.
2. **Read selectively:** Each phase lists its Reference sections — read only those parts of the PRD, vision, and design docs, not the whole files.
3. **Let it work:** The agent finds the first unchecked task, implements it, and marks it `- [x]`.
4. **One session = one phase (ideally):** Aim to complete a phase per session; resume from the last unchecked task if you stop.
5. **Push a PR for review:** When a phase is complete, push it and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review before starting the next phase.
6. **Need more context?** If a task references a section not in the phase's list, read just that section on demand.

### Session Tips

- **Don't read everything** — use the Reference sections to stay focused and save context.
- **Don't skip tasks** — they're ordered to resolve dependencies.
- **Test offline** — the core path must work in airplane mode; verify there after every phase.
- **Review before moving on** — merge the phase PR before starting the next.
- **Update the status line** — after completing tasks, update `**Status:** X/58 tasks complete` and `**Current Phase:**` in the header.

### Phase Review Workflow

After each phase, create a branch `phase-{N}/{slug}`, commit, push, and open a PR titled `Phase {N}: {Title}` with the goal and completed task count. Let an automated review agent (CodeRabbit recommended — free for open source) review it. Address feedback on the same branch, re-review, and merge only when clean. If you don't use GitHub, you can skip PR review — but you lose automated catching of bugs and security issues; the per-task verification still applies.

**Resuming after a break:**

**Fixing an issue:**
